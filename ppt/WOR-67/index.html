Engineering Preview

What is Symphony?

"Symphony turns project work into isolated, autonomous implementation runs, allowing teams to manage work instead of supervising coding agents."

The Problem It Solves

Traditional AI-assisted coding requires constant human supervision — you manage the agent, not the work. As complexity scales, this becomes a bottleneck.


Symphony inverts this: you manage work items (tickets, stories) while agents figure out the implementation autonomously.

What It Is

  • A daemon service (runs continuously)
  • Monitors a project board (e.g. Linear)
  • Spawns coding agents per ticket
  • Each agent works in an isolated workspace
  • Provides proof of work (CI, PR, video)
  • Safely lands PRs once accepted

The Paradigm Shift

Before Symphony

Managing Agents

  • You watch the agent work
  • You debug when it gets stuck
  • You approve each action
  • One agent, one human
  • Hard to scale beyond a few concurrent tasks

After Symphony

Managing Work

  • You write tickets with clear AC
  • Symphony dispatches agents
  • You review PRs (not agent sessions)
  • Many agents, one human
  • Scales with your ticket queue

The shift is from "supervise AI" → "manage work". Your attention moves up the stack: from implementation details to product decisions.

7-Component Architecture

📄

Workflow Loader

Reads & hot-reloads WORKFLOW.md. YAML front matter + Markdown prompt body. Dynamic config without restarts.

⚙️

Config Layer

Typed getters with defaults. Supports $VAR indirection for secrets (e.g. $LINEAR_API_KEY).

📊

Issue Tracker Client

Fetches active/terminal issues from Linear (GraphQL). Normalizes to a stable domain model.

🧠

Orchestrator

Single authority over in-memory state. Dispatch eligibility, retry with exponential backoff, reconciliation.

📁

Workspace Manager

Per-issue isolated directories. Lifecycle hooks: after_create, before_run, after_run, before_remove.

🤖

Agent Runner

Launches coding agent subprocess. Line-delimited JSON-RPC protocol. Streams updates back to orchestrator.

🔍

Status Surface & Logging

Optional HTTP server (GET /api/v1/state). Structured logging with issue_id, session_id context.

The Orchestrator — Decision Engine

Poll-and-Dispatch Loop (every 30s)

  1. Reconcile running agents vs. tracker state
  2. Fetch candidate issues (active states)
  3. Sort by priority → date → identifier
  4. Dispatch eligible issues to agents
  5. Report to observability consumers

Issue Eligibility Rules

  • State in active_states
  • Not already running or claimed
  • Concurrency slots available
  • If Todo: no un-resolved blockers

Retry Strategy

  • Clean exit: 1-second continuation retry
  • Error exit: Exponential backoff min(10s × 2ⁿ, 5min)
  • Stall: Agent inactive → kill + retry

Issue States in Orchestrator

  • Unclaimed → Claimed → Running
  • Running → RetryQueued (on failure)
  • Running → Released (terminal state)

WORKFLOW.md — The Policy File

WORKFLOW.md is the heart of Symphony. It's version-controlled alongside your code — the workflow evolves with the codebase.


File Structure

---
# YAML front matter: config
tracker:
  kind: linear
  api_key: $LINEAR_API_KEY
  project_slug: my-team
  active_states: [Todo, In Progress]

polling:
  interval_ms: 30000

agent:
  max_concurrent_agents: 3

codex:
  command: codex app-server
  turn_timeout_ms: 3600000
---

# Markdown body: the agent's prompt
You are working on ticket {{ issue.title }}.

## Instructions
...read the ticket, write code, create PR...

Why WORKFLOW.md?

  • One file controls the agent's entire behavior
  • Hot-reload: change behavior without restarting Symphony
  • Team-reviewable: the "system prompt" is a PR
  • Per-repo: monorepo can have per-app workflows

Template Variables

  • {"{{ issue.title }}"} — ticket title
  • {"{{ issue.description }}"} — full description
  • {"{{ attempt }}"} — retry count
  • Strict rendering — unknown vars = hard fail

Workspace Isolation — Safety by Design

Every ticket gets its own isolated workspace directory. This is not optional — it's a core safety invariant.


3 Invariants (Must Enforce)

  1. Agent runs only in its workspace path — cwd == workspace_path validated before launch
  2. Workspace stays inside root — absolute paths checked, escape attempts rejected
  3. Workspace key is sanitized — [^A-Za-z0-9._-] replaced with _

Lifecycle Hooks

  • after_create — setup, install deps (fatal on failure)
  • before_run — pre-agent checks
  • after_run — cleanup (non-fatal)
  • before_remove — pre-deletion (non-fatal)

Benefits

  • No cross-ticket contamination
  • Failed agents don't corrupt other work
  • Workspace reuse for fast retries
  • Clean shutdown: terminal issues → workspace deleted

Prerequisite: Harness Engineering

Symphony works optimally in codebases that have adopted harness engineering — an approach to managing AI coding agents.
🔍

Legibility

Code and architecture an AI agent can navigate without human guidance. Clear naming, obvious file structure, minimal magic.

🏗️

Architectural Enforcement

Linting, type checks, and tests that run in CI — so agents know immediately if they broke something.

🧹

Entropy Management

Small, well-scoped tickets. Clear Acceptance Criteria. Avoid AI-generated sprawl that future agents can't navigate.

A codebase that humans can navigate quickly is also a codebase agents can navigate autonomously. They share the same prerequisites.

Proof of Work — How Agents Demonstrate Completion

Symphony agents don't just say "I'm done." They provide verifiable evidence through multiple channels.


Evidence Channels

  • 🟢 CI Status — all checks green
  • 📋 PR Review Feedback — agent addresses review comments
  • 📊 Code Complexity Analysis — no scope creep
  • 🎥 Walkthrough Video — screen recording of the feature working

Why This Matters

You don't need to watch the agent work. You review the output — just like reviewing a PR from a junior engineer.


Safe Landing

Once a PR is approved, Symphony merges it autonomously. The human decision is "approve / request changes" — not when/how to merge.

poll-linear.mts — The Orchestrator

My orchestrator implementation — runs permanently as a local daemon.


Core loop

  1. Fetch all configured boards from Linear
  2. Find eligible tickets (assigned to me, in Todo/In Progress)
  3. Spawn Claude Code agents (up to max concurrent)
  4. Monitor agents, handle exits / reworks
  5. Render live dashboard
  6. Sleep 30s, repeat

Extra features (beyond spec)

  • Rate-limit detection → sleep-in-place until window expires
  • Multi-board: one daemon, many Linear teams
  • Interactive CLI (resume <id>, help)
// Core dispatch loop
while (true) {
  // Rate-limit: sleep in-place (not re-poll)
  if (rateLimitPausedUntil) {
    const ms = rateLimitPausedUntil - Date.now();
    await sleep(ms);
    rateLimitPausedUntil = null;
  }

  await poll();  // fetch + reconcile + dispatch

  await sleep(POLL_INTERVAL_MS);
}

async function poll() {
  for (const board of boards) {
    const todos = await fetchByState(board, 'Todo');
    for (const ticket of todos) {
      if (!isEligible(ticket, board)) continue;
      if (runningAgents.has(ticket.identifier)) continue;
      if (runningAgents.size >= MAX_CONCURRENT) break;

      await moveToInProgress(board, ticket);
      spawnAgent(ticket, board, 'fresh');
    }
  }
}

run-ticket.sh — Workspace + Agent Runner

Per-ticket script — called by the orchestrator for each new ticket.


1. Workspace = Git Worktree

Instead of a plain directory, each ticket gets a git worktree:

  • Isolated branch: feat/WOR-67-...
  • Symlinked node_modules (instant setup)
  • Full git history available to agent
  • Multiple worktrees share one repo → disk efficient

2. Agent = Claude Code

claude --dangerously-skip-permissions with WORKFLOW.md injected as the startup prompt. The agent reads the ticket, explores code, implements, validates, and creates a PR.

#!/usr/bin/env bash
# Simplified run-ticket.sh

BRANCH="feat/${TICKET_ID}-${SLUG}"
WORKTREE="$WORKTREES_DIR/$BRANCH"

# 1. Create isolated workspace (git worktree)
git worktree add "$WORKTREE" \
  -b "$BRANCH" origin/master

# 2. Symlink node_modules (instant)
ln -s "$REPO_ROOT/node_modules" \
  "$WORKTREE/node_modules"

# 3. Build prompt from WORKFLOW.md template
PROMPT=$(envsubst < ~/.claude/WORKFLOW.md)

cd "$WORKTREE"

# 4. Launch Claude Code agent
claude \
  --dangerously-skip-permissions \
  --print "$PROMPT" \
  >> "logs/symphony-$TICKET_ID.log" 2>&1

My WORKFLOW.md — The Agent's Playbook

Lives at ~/.claude/WORKFLOW.md in claude-home. Version-controlled, reviewable as a PR.


7 Steps (from the workflow)

  1. Read & Plan — fetch ticket, create workpad
  2. Setup Worktree — already done by runner
  3. Implement — read code first, write in phases
  4. Validate — lint, tests, type-check
  5. Rebase + PR — rebase onto master, create PR
  6. Submit for Review — evidence to workpad
  7. Merge — land skill, exit

Skills System

Each step references a Markdown skill file in ~/.claude/skills/. The agent reads the skill at runtime.


Skills = reusable, version-controlled "sub-instructions". A better create-pr skill improves every future ticket.

Linear Workpad

A persistent comment on the ticket. The agent's scratchpad: plan, AC checkboxes, progress notes. Survives daemon restarts.

What works well

  • Small, well-defined tickets with clear AC
  • Frontend features following existing patterns
  • Bug fixes with clear reproduction path
  • Chores: config changes, dependency updates

What still needs humans

  • Ambiguous or under-specified tickets
  • Novel architecture decisions
  • Multi-team product decisions
  • Debugging flaky E2E tests

Key Learnings

1. Ticket quality is the real bottleneck

A ticket with clear Acceptance Criteria gets solved autonomously. A vague ticket gets reworked 3× and wastes agent time. Symphony amplifies your process — good or bad.

2. WORKFLOW.md is your product

Every improvement to ~/.claude/skills/ instantly improves all future tickets. Treat it like production code — review it, test it, iterate.

3. Git worktrees = perfect isolation

Zero overhead isolation. Multiple agents, one repo, no conflicts. The spec's workspace requirement maps perfectly to git worktrees.

4. Human review is a feature

The PR review step catches the 20% of cases where the agent went in the wrong direction. Don't skip it — it's what makes the system trustworthy.

5. Sleep-in-place for rate limits

Sleep inside the current poll tick until the window expires — don't return early. This keeps agent session context alive and allows immediate resumption.

6. The spec is the starting point

I deviated from the spec (Claude Code vs Codex, TypeScript vs Elixir, git worktrees vs plain dirs). The spec gives you the design — you own the implementation.
